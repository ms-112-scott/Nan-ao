# 南澳知識圖譜計畫 — 詳細規劃 v2

> **v2 變更摘要**：對齊真實 xlsx 資料 + 確認管線（Sheets → GAS → GitHub → Frontend）。
> v1 中假設的 parent_id、3 層樹、新建 schema 全部修正為「以現有欄位為主，最小加欄」。
> v1 文件保留作對照，本文為新版基準。

## 0. 真實資料盤點（v2 新增）

### 0.1 資料規模

| 串流（研究者） | nodes 列數 | links 列數 | 主題範疇                        |
| -------------- | ---------- | ---------- | ------------------------------- |
| 雅柔           | 519        | 1003       | 部落遷徙史、文化健康站          |
| 竺庭           | 1012       | 1006       | 段木香菇產業、林業              |
| 文化活動       | 1016       | 959        | 大南澳事件、開山撫番、清/日治史 |
| 總計畫         | 141        | 182        | 陽明交大 USR 教案、健康飲食     |
| **合計**       | **2688**   | **3150**   | —                               |

去重合併後估計：**唯一節點 1500–2000、唯一邊 2500–3000**。已超過原規劃 1000 節點上限，效能策略要加強（見 §13）。

### 0.2 真實 schema（重要）

**nodes 欄位**（你目前用的）：

```
id  |  node_Group  |  info  |  start_year  |  end_year  |  Lon  |  Lat
```

**links 欄位**：

```
Node_A  |  Node_B  |  label  |  info  |  Date
```

**輔助表**：

- `nodes_color`：node_Group → 顏色（hex）的對照表
- `links_color`：link label → 顏色的對照表（已整理 79 種 label）
- `missing_node`：GAS 驗證輸出，列出 link 中引用但 nodes 表未定義的 ID

### 0.3 觀察到的 node_Group 多樣性

從 4 個 sheet 抽樣可見的 node_Group 值（非完整）：

| 大類（v2 新增聚合） | 既有 node_Group                              |
| ------------------- | -------------------------------------------- |
| 人物                | `人物`, `核心人物`                           |
| 組織                | `政府機關`, `非營利組織`, `私人企業`, `大學` |
| 地景與聚落          | `部落`, `地理位置`, `古道`（推估）           |
| 事件                | `大事件`, `事件`, `政策`（推估）             |
| 物質文化            | `物件`, `食物`, `工藝`（推估）               |
| 文獻                | `論文`, `報告`, `紀錄片`（推估）             |
| 計畫                | `計畫`, `課程`                               |

**問題**：node_Group 是自由文字，4 位研究者會有同義不同詞（如「核心人物」vs「人物」、「機構」vs「組織」）。
**對策**：在 `nodes_color` 加一欄 `meta_group`，把細分群歸到 6–7 個一級分類。前端用 meta_group 做篩選與視覺分群，用 node_Group 做圖例顏色。

### 0.4 觀察到的 link label 多樣性

`links_color` 已收 79 種 label，例如：空間關係、人際關係、政策推動、行動參與、文化嚮導、開闢、紀念、合作夥伴、負面影響、學習工具、研發單位…

**對策**：

- 不在前端硬分「階層/跨越/時序」，而是把 label 在 `links_color` 加一欄 `meta_relation`，歸到 5 種：
  - `spatial`（空間關係、開闢、衍生分布）
  - `social`（人際關係、合作、師承）
  - `causal`（政策推動、負面影響、引發）
  - `creative`（內容設計、研發、創作）
  - `documentary`（紀念、記錄、引用）
- 前端篩選時可選 meta_relation，顯示時仍用 link 自己的 label 顏色

## 1. 與 v1 的關鍵差異

| -------- | ---------------------------- | ------------------------------------------------------------------------------------ |
| 階層     | parent_id + 3 層樹 + 摺疊    | **無 parent_id**；改用 meta_group → node_Group → 節點的「視覺分群」（force cluster） |

## 2. 技術選型（v2）

### 2.1 前端

- **React 19 + Vite + Tailwind v4**（沿用 Zhudong 工程習慣）
- **D3 v7 + Canvas**（強制；2000 節點 SVG 必崩）
- **d3-quadtree** 命中測試
- **DOM overlay** 渲染節點 label（只渲染視窗內 + zoom > 0.8）— 紙質感襯線字 Canvas 渲染品質太差
- **不用** react-router（單頁，URL query string 即可）

### 2.2 資料管線（你選定的方案，細化）

```
┌─────────────────┐    ┌──────────────┐    ┌────────────┐    ┌──────────┐
│ Google Sheets   │───▶│ Apps Script  │───▶│  GitHub    │───▶│ Frontend │
│ (4 contributors)│    │  (build.gs)  │    │  data.json │    │  (Pages) │
└─────────────────┘    └──────────────┘    └────────────┘    └──────────┘
   多人即時編輯           時間觸發/手動         版本歷史          快取友好
```

**Apps Script 職責**（取代 Zhudong 的「前端即時打 API」模式）：

1. 讀所有 `nodes_*` / `link_*` 工作表
2. 合併同 ID 節點（info 串接、衝突偵測、塗紅）
3. 補完缺漏節點警示（塗黃 + 寫入 `missing_node` 表）
4. 套用 meta_group / meta_relation 對照
5. 產出標準化 `data.json`
6. 透過 GitHub REST API（PAT 存在 Apps Script Properties）commit 到 repo `data/data.json`
7. commit message 含時間戳與節點/邊統計

**前端職責**：

- 載入 `https://cdn.jsdelivr.net/gh/<user>/Nanao@latest/data/data.json`
- 用 jsDelivr 加上 commit hash 確保即時生效（如 `@<sha>/data/data.json`）
- 完全靜態，無 GAS 依賴，可離線快取

**這個架構優勢**：

- 研究者繼續編 Sheets（零學習曲線）
- JSON 進版控（可回溯、可比對任意兩版差異）
- 前端無 GAS 配額焦慮、無 CORS 問題、SLA 高
- jsDelivr 全球 CDN 快取，Pages 退化成靜態主機

**觸發方式**（兩擇一）：

- A. Apps Script 時間觸發器，每天凌晨 03:00 自動跑
- B. Sheet 中放一個「發布」按鈕（onEdit 或 menu），文史工作者改完手動觸發

→ 建議 **A + B 並行**：日常自動每日同步；發現重大修正可手動推送。

### 2.3 託管

- GitHub Pages，路徑 `https://<user>.github.io/Nanao/`
- 自訂網域可後加

### 2.4 狀態管理

- URL query string 為單一真實來源
- 自製 `useUrlState` hook 雙向同步

## 5. 視覺語言（v2 — 處理顏色衝突）

你選了「紙質感」，但既有 `nodes_color` 是高彩度（#E63946 朱紅、#FF8E8E 粉紅、#4CC9F0 亮藍、#F4A261 橘）。三條路：

### 路線 A：完全改 sheet 顏色（推薦）

更新 `nodes_color` / `links_color` 為紙質感色票（見 §5.2）。
優點：sheet 與前端視覺一致，研究者打開 sheet 也是溫潤調性。
缺點：要說服研究者重新挑色。

### 路線 B：前端覆蓋

忽略 sheet 顏色，前端寫死紙質感 palette。
優點：不動研究者的工作流。
缺點：sheet 的顏色變成「死資料」，研究者疑惑。

### 路線 C：放棄紙質感，沿用既有色

換個視覺基調（如資訊圖表清爽風 — 白底 + 高對比）。
優點：零衝突。
缺點：失去你想要的文史氛圍。

→ 我推 **A**。多花一次溝通，整體一致性最好。下方 §5.2 就是建議的紙質色票（你可以直接給文史工作者改 sheet）。

### 5.3 字體（沿用 v1）

- 中文標題：思源宋體 Heavy
- 中文內文：思源宋體 Regular
- 中文副標：思源黑體 Light
- 西文/數字：EB Garamond
- 自託管在 `public/fonts/`

### 5.4 節點造型

- 圓形為主，依 meta_group 微調
  - 人物：實心圓
  - 組織：圓 + 方框外圈
  - 地景：圓角矩形
  - 事件：六邊形
  - 文化：菱形
  - 文獻：書頁形
  - 計畫：圓 + 虛線外圈
- 大小：`8 + (importance ?? 3) * 4` px
- 突破點：1.5x 半徑金色光暈 + 右上角金色八角星

### 5.5 邊樣式

- 依 `meta_relation` 取線型（實/虛/點/箭頭）
- 顏色取自 `links_color` 的 color 欄位
- 寬度：`(weight ?? 2) * 0.7` px
- Canvas 繪製時加 perlin noise（±0.5px）做手繪感

### 5.6 紙質感背景

- 1024×1024 細噪點米紙紋理 PNG，平鋪 30% 透明
- 角落內陰影

## 7. 群聚力導向佈局細則（v2 取代 v1 階層樹）

### 7.1 為何不用 d3-hierarchy

真實資料無 parent_id；強行套階層會做出錯誤的關係結構。

### 7.2 群聚 force 設計

```javascript
// 偽碼，僅作說明
const groupCenters = {
  人物: [-300, -200],
  組織: [300, -200],
  地景與聚落: [-400, 0],
  事件: [400, 0],
  物質文化: [-300, 200],
  文獻: [300, 200],
  計畫與行動: [0, 300],
};

simulation
  .force("x", d3.forceX((d) => groupCenters[d.meta_group][0]).strength(0.15))
  .force("y", d3.forceY((d) => groupCenters[d.meta_group][1]).strength(0.15))
  .force("charge", d3.forceManyBody().strength(-200))
  .force(
    "link",
    d3
      .forceLink(links)
      .distance((d) => (d.meta_relation === "spatial" ? 50 : 120)),
  )
  .force(
    "collide",
    d3.forceCollide().radius((d) => nodeRadius(d) + 4),
  );
```

### 7.3 視覺效果

- 同類節點聚成一團（如所有人物在左上）
- 跨類連結把團拉近，形成「群島」式視覺
- 比 v1 的同心圓階層更貼近真實資料的混合性

### 7.4 「聚焦單一群」模式

- 點圖例的 meta_group 色塊：
  - 該群節點放大、其他縮小
  - 相關邊高亮、無關邊淡化
- 適合一次只看一個面向（如「先看所有人物」）

## 9. 篩選管線（v2）

依序套用，useMemo 快取：

1. **時間範圍**（滑桿）：節點 start/end 與區間有交集；無年份視為永恆通過
2. **meta_group**（圖例）：勾選的 meta_group
3. **meta_relation**（關係 toggle）：勾選的 meta_relation
4. **重要性閾值**（可選）
5. **突破點 only**（toggle）
6. **搜尋詞**（即時）
7. 邊：source/target 都通過才保留

**效能關鍵**：1500–2000 節點 + 3000 邊，每次篩選都重跑 force 太慢。
策略：篩選只控**顯示與否**，不重啟 simulation；force 只在第一次跑到穩定，之後 simulation.stop()。

## 12. 資料管線細節（v2 — 你選定的方案）

### 12.1 GAS `build.gs` 流程

```
1. 讀所有 nodes_* 與 link_* 工作表
2. 讀 nodes_color, links_color 對照表
3. 對所有 nodes 做合併：
   - 同 id → info 串接（用「；」分隔，標註來源 sheet）
   - node_Group 衝突 → 取第一個出現的，並塗紅標記
4. 對所有 links 做去重（同 source-target-label 視為一條）
5. 套用 meta_group / meta_relation 對照
6. 跑驗證（§4.7 規則）
7. 把驗證結果寫回 missing_node 表 + 對應儲存格上色
8. 組成 data.json：
   {
     "version": "<commit_hash_or_timestamp>",
     "generated_at": "2026-05-11T03:00:00Z",
     "stats": { "nodes": 1872, "links": 2934, "breakthroughs": 12 },
     "meta_groups": [{ "id": "人物", "color": "#4A6B5B" }, ...],
     "meta_relations": [{ "id": "spatial", "color": "#8B5A2B" }, ...],
     "nodes": [...],
     "links": [...]
   }
9. 透過 GitHub REST API 用 PAT commit 到 repo
   PUT /repos/<user>/Nanao/contents/data/data.json
   commit message: "data: 1872 nodes, 2934 links, 12 breakthroughs (auto)"
```

### 12.2 GAS 設定步驟

1. 在 Sheet 中 Extensions → Apps Script
2. 把 `apps_script/*.gs` 複製進去
3. Properties Service 設兩個值：
   - `GITHUB_PAT`：個人 access token（只給 contents:write 權限）
   - `GITHUB_REPO`：`<user>/Nanao`
4. 設時間觸發器：每天 03:00 跑 `buildAndPush()`
5. Sheet 加自訂選單「📤 立即發布」呼叫同函式

### 12.3 衝突解決流程（多研究者場景）

當「武塔」同時出現在雅柔、竺庭兩個 nodes 表：

```
情境 1：node_Group 一致
  → 直接合併 info：「[雅柔] 武塔的早期遷徙... ; [竺庭] 武塔的香菇產業...」

情境 2：node_Group 不一致（例如雅柔填「部落」、竺庭填「地理位置」）
  → 取第一個出現的（雅柔）
  → 在兩處儲存格塗紅
  → 在 missing_node 表記錄衝突
  → 寄 email 通知研究者協調（optional）
```

## 13. 效能策略（v2 — 因應 1500–2000 節點）

| 項目     | 策略                                                |
| -------- | --------------------------------------------------- |
| 渲染     | Canvas + requestAnimationFrame                      |
| 命中測試 | d3-quadtree 每 tick 重建                            |
| Label    | viewport culling + zoom > 0.8 才渲染                |
| Force    | 跑到 alpha < 0.005 後 simulation.stop()，篩選不重啟 |
| 圖片     | 預生成 200x200 縮圖，原圖 lazy load                 |
| JSON     | gzip + jsDelivr CDN 快取                            |
| 篩選     | useMemo + Set 快查                                  |
| 移動端   | 預設只顯示 importance >= 3 的節點                   |
| 邊       | zoom < 0.5 時暫不渲染邊（只顯節點群）               |

效能目標：

- 初始載入 < 3 秒
- 拖動 60fps
- 篩選切換 < 300ms
